## Design Context

### Users
Solo developer using Forge as a daily driver to orchestrate 5-20+ parallel Claude Code / Codex CLI terminals. The user fantasy is **code craftsman** — someone who works with intensity across multiple agents simultaneously, building software (Forge itself, web apps, a game, a knowledge base). The app wraps real PTY terminals; there is no custom chat UI. Everything is built around the terminal as the core interaction surface.

### Brand Personality
**Intensity. Focus. Craft.**

Forge evokes focused intensity and workstation confidence — the feeling of sitting at a well-tuned command center where every surface gives you exactly the information you need. The emotional arc is flow state: the UI should carry the user into deep focus and reward sustained engagement through escalating visual feedback, not punish breaks.

### Aesthetic Direction
- **Digital code forge** — not a skeuomorphic blacksmith's forge, but a precision-engineered digital workspace
- **Inspired by game menu design**, particularly League of Legends' Hextech UI: that same feeling of crafted, luminous surfaces with purposeful glow, sharp geometry, and layered depth — but applied to a developer tool
- **Dark mode only.** Deep blacks (#090909), minimal chrome, bright accents reserved for status and heat
- **Heat metaphor is central.** Four themes (Forge/amber, Frost/cyan, Void/purple, Spore/green) each "heat up" as users string together prompts on streak/cooldown timers. This is not punishing — there are no visible timers or progress bars. The escalation is purely ambient: background shifts, glow intensifies, borders warm. The point is to amplify flow, not measure it
- **Anti-references:** Generic dashboards, consumer-friendly rounded apps, skeuomorphic textures, overly "cute" or playful aesthetics. This is a power tool that looks like one

### Design Principles

1. **Serve the terminal.** The terminal is the primary interaction surface. Every UI element exists to support orientation across terminals — never to compete with them. Chrome should recede; status should be glanceable.

2. **Earn every pixel.** No decorative elements. Every glow, border, and color shift encodes real information (status, heat stage, group membership). If it glows, it means something. Inspired by Hextech: ornate but purposeful.

3. **Amplify flow, don't measure it.** The heat system rewards sustained engagement through ambient visual escalation. No timers, no scores, no progress bars. The user should feel the heat rising without consciously tracking it. The reward is the aesthetic itself.

4. **Quiet until urgent.** Default state is calm and receded (slate, low contrast, minimal glow). Visual intensity scales with actual urgency — a waiting terminal at high heat should pull your eye; an idle terminal at cold should disappear. The dynamic range between cold and meltdown is the design's expressive power.

5. **Craft in the details.** Transitions should feel intentional and physical — not CSS defaults. Hover states, panel reveals, status changes, and menu animations should all feel like they belong to the same precisely-engineered system. Game-quality micro-interactions, not web-app afterthoughts.

### Technical Design Notes
- Custom CSS with 45+ CSS variables, no framework
- 4 themes × 6 heat stages, dynamically applied via React + CSS custom properties
- Typography: Segoe UI (UI), IBM Plex Mono (code), 13px base
- Spacing: 2-16px scale, sharp corners (0-3px radius), fixed sidebar (200px)
- Motion: 0.1s hover, 0.35s status transitions, 1.5s breathing pulses, 240-300ms flash cues
- Key files: `src/utils/themes.js`, `src/styles/global.css`, `src/App.jsx`
