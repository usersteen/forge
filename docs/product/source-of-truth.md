# Forge Source Of Truth

Last updated: 2026-04-09
Status: Draft for first public launch

## Purpose

This document is the canonical messaging and product-positioning reference for Forge.

Use it to keep the following in sync:

- landing page
- README
- in-app welcome flow
- info panel
- guided tour
- GitHub release copy

This is not a changelog and not a feature backlog. It defines what Forge is, how it should feel, what it should claim publicly, and how each surface should express that truth.

## Brand Core

### Product idea

Forge is a desktop workstation for parallel AI coding.

It is not primarily about "managing agents." It is about giving the user a command surface that feels powerful, legible, and alive while they work across multiple coding terminals.

### Player fantasy

Your computer becomes a code forge.

You are not buried in terminals or babysitting tools. You are seated at a command station, moving between active workstreams with clarity and control. The interface supports intensity without chaos. It rewards engagement with atmosphere, motion, and heat, but it does not pressure the user with overt productivity mechanics.

Forge should make the user feel:

- in command
- oriented
- powerful
- focused
- rewarded for staying in flow

### Emotional promise

Forge turns parallel AI coding into a crafted workstation experience.

### Practical promise

Forge helps you run multiple AI coding terminals in parallel, organize them by project, see which ones need attention, and stay in flow without losing context.

### What Forge is not

Forge is not:

- a generic agent dashboard
- a productivity timer
- a project management tool
- a full IDE
- a chat wrapper pretending to be a terminal
- a punishing gamification system

## Positioning

### Primary positioning statement

Forge is a desktop command workstation for parallel AI coding.

### Strategic differentiation

Other products in this category often lead with orchestration, monitoring, or feature breadth.

Forge should lead with experience.

The point is not just that you can run multiple coding agents. The point is that Forge makes parallel AI coding feel deliberate, readable, and powerful. It gives the user a workstation identity, not just a utility layer.

### Positioning summary

Forge wins on:

- experiential quality
- strong emotional identity
- terminal-first command feel
- ambient reward systems that reinforce focus

Forge does not need to win by claiming the widest feature set.

## Heat Philosophy

### Canonical meaning of heat

Heat is an ambient reward system.

It reflects active engagement and carries the user deeper into a focus state. As the user responds to waiting work and keeps momentum, the interface warms up. The reward is atmospheric: glow, motion, color, and presence.

### What heat is not

Heat is not:

- a countdown
- a visible score
- a punishment mechanic
- stress-inducing urgency
- a productivity metric for external judgment

### Messaging guidance for heat

When describing heat:

- describe it as atmosphere, momentum, rhythm, or reward
- connect it to flow state
- emphasize that it is ambient and non-punishing

Do not describe heat as:

- pressure
- urgency
- performance scoring
- gamified accountability

## Launch Truth

### Core launch story

Forge is ready to launch as a focused workstation for parallel AI coding.

The launch story is not "everything for everyone." It is a clear operating model with a distinctive feel.

### Core launch features

These are the first-class launch features. They should be the backbone of public messaging.

- Project-based workspace organization
- Multiple terminal tabs per project
- First-class Claude Code and Codex tab creation
- Server tabs for long-running processes
- Automatic terminal status detection
- Status visibility across tabs and projects
- Repo binding and project explorer
- Side-by-side document viewing
- Markdown editing in the document pane
- Heat-based visual feedback
- Theme system
- Audio alerts for waiting tabs
- Session persistence across restarts

### Secondary launch features

These can be mentioned, but should not dominate top-level messaging.

- keyboard shortcuts
- favorite repos
- configurable sound volume
- configurable heat timing
- project menu detail modes
- theme-specific particle behavior

### Current limitations

These are acceptable to acknowledge in practical surfaces like the README.

- Forge is a focused workstation, not a full IDE
- Onboarding and launch messaging are still being tightened
- Feature depth is intentionally narrower than some broader orchestration tools

Do not frame these limitations apologetically. State them plainly.

## Surface Rules

Each touchpoint has a different job. Do not make every surface say the same thing in the same way.

### Landing page

Job:

- sell the feeling
- define the category on Forge's terms
- make the workstation fantasy concrete

Rules:

- lead emotional first
- prove the feeling with practical product behavior
- keep launch scope tight
- do not oversell breadth

Should emphasize:

- command
- flow
- heat as ambient reward
- terminal-first orientation

### README

Job:

- explain clearly what Forge does
- help someone install it and decide whether it fits their workflow

Rules:

- lead practical first, with a hint of brand voice
- keep feature descriptions concrete
- acknowledge limitations without apology

### Welcome flow

Job:

- orient a first-time user
- get them to first success quickly

Rules:

- less brand lore, more immediate setup
- explain the mental model in one breath
- point toward first useful action

The welcome flow should answer:

- what Forge is
- what a project is
- what a tab is
- what to do first

### Info panel

Job:

- explain the operating model
- serve as the compact in-app reference

Rules:

- practical and structured
- no vague marketing lines
- only mention things the user can find in the app right now

### Guided tour

Job:

- teach interaction

Rules:

- explain visible UI only
- keep copy short
- do not carry brand positioning burden
- do not promise docs or concepts that are not present in the current UI

### GitHub release notes

Job:

- explain what changed in this release

Rules:

- release notes are deltas, not the core product story
- do not reintroduce conflicting positioning

## Tone

### Desired tone

- emotionally charged but controlled
- confident
- crafted
- practical beneath the surface
- intense without being theatrical

### Tone boundaries

Do:

- write with precision
- use short, sharp statements
- let the product feel intentional
- make emotion serve clarity

Do not:

- sound like a generic SaaS tool
- sound like a game trailer
- over-explain the metaphor
- make claims broader than the current product

## Copy Tests

Any new public-facing copy should pass these tests:

### 1. Forge test

Could this line belong to any generic agent tool?

If yes, it is too weak.

### 2. Truth test

Can a user verify this claim in the current product?

If not, cut it or soften it.

### 3. Role test

Does this surface say what its specific surface is supposed to say?

If the welcome modal is trying to do the landing page's job, or the guided tour is trying to do the README's job, the copy is wrong even if the words are good.

### 4. Heat test

Does the language describe heat as reward and atmosphere rather than pressure and judgment?

If not, rewrite it.
