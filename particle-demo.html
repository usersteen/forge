Particle System V2 -- Research & Design Reference

Why V2?

V1 uses CSS @keyframes on DOM <span> elements. Each particle is a real DOM node animated by the browser's CSS engine. This worked well up to ~80 particles but hits hard limits:

  • Single color per particle -- particles use background: var(--heat-4) and never change color during their lifetime
  • Fixed motion paths -- CSS keyframes define predetermined curves; no organic/physics-based movement
  • ~80 particle cap -- DOM elements are expensive (layout, paint, composite per node)
  • No compositing control -- can't do additive blending on DOM spans; glow is faked via box-shadow
  • Theme rigidity -- each theme needs ~150 lines of separate @keyframes blocks in global.css

V2 moves to Canvas 2D + requestAnimationFrame. This is the standard approach for real-time 2D particle effects in browsers. One canvas element replaces dozens of DOM nodes.

Research Summary

Rendering Approach

We evaluated three approaches:

ApproachParticle CapComplexityDependencies
CSS/DOM (v1)~80LowNone
Canvas 2D (chosen)~500-1000MediumNone
WebGL (PixiJS etc)10,000+HighPixiJS ~400KB

Canvas 2D is the sweet spot: 10x the particle budget of CSS with zero dependencies. WebGL would be overkill for our target of 200-500 particles and adds a heavy library dependency. Canvas 2D at this count runs comfortably at 60fps even on integrated GPUs.

Object Pooling

The ParticlePool class pre-allocates all particle objects at init time. During animation, particles are recycled from the pool -- no new calls, no garbage collection pressure. This pattern comes from game development (Game Programming Patterns by Robert Nystrom) and is critical for smooth animation: GC pauses can spike frame times from 16ms to 50ms+.

Implementation: flat array of particle structs with active flag, rotating cursor for O(1) amortized allocation. Pool sized at 600 (headroom above peak active count of ~200-400).

Force-Based Physics

V1 particles follow hardcoded CSS keyframe paths. V2 uses composable force functions applied per-frame:

  • Gravity -- constant acceleration; negative = rise (forge embers), positive = fall (ice crystals)
  • Turbulence -- 2D simplex noise sampled at each particle's position. Each particle has a unique seed offset so they sample different parts of the noise field, creating organic diversity. Three parameters: frequency (spatial scale), speed (temporal change rate), amplitude (force strength)
  • Wind -- constant or time-varying directional force. Grass uses Math.sin(time * 0.7) * 5 for a swaying breeze
  • Drag -- velocity damping each frame (v *= 1 - damping * dt). Prevents infinite acceleration, gives particles a natural deceleration feel

Forces compose additively: the emitter config lists which forces to apply and their parameters. This is far more flexible than CSS keyframes -- changing one force parameter alters the entire feel without rewriting animation curves.

Simplex Noise

The turbulence force uses 2D simplex noise (Stefan Gustavson's public domain algorithm, ~40 lines inline). Simplex noise produces smooth, natural-looking pseudo-random values that vary continuously over space and time. This is the same technique used in procedural terrain generation, fluid simulations, and professional VFX particle systems. It's computationally cheap (~10M evaluations/sec in JS), making it practical for per-particle-per-frame sampling.

Color Over Lifetime

V1: single color (var(--heat-4)). V2: multi-stop gradient interpolated as particle ages.

Color stops are derived from the existing heatColors arrays in themes.js. We chose this deliberately: the heat colors were carefully selected for thematic coherence and contrast with status dots. The particle color ramp walks from heatColors[5] (hottest, birth) down to heatColors[0] (coolest, death), so particles literally "cool down" through the same palette used throughout the UI.

// Example: forge particle cools through its own heat palette
t=0.0  #ff4d4d (heat-5, birth)  -- brightest
t=0.15 #ef4444 (heat-4)
t=0.35 #dc2626 (heat-3)
t=0.55 #ea580c (heat-2)
t=0.75 #f59e0b (heat-1)
t=1.0  #475569 (heat-0, death)  -- coolest

Interpolation is linear RGB with lerpColorStops(). We considered CIELAB for perceptual uniformity but linear RGB is fine for these warm/saturated palettes and avoids the conversion overhead per particle per frame.

Additive Blending

ctx.globalCompositeOperation = 'lighter' makes overlapping particles accumulate light naturally. Where multiple particles cluster, the glow intensifies automatically -- no extra bloom pass needed. This is how most 2D game particle systems achieve glow effects cheaply.

Each particle gets two draw calls: a large, faint glow pass (size * glowRadius, alpha * glowAlpha) drawn first, then the sharp core pass on top. The glow pass creates soft halos; additive blending makes them build up organically where particles overlap.

Theme-Specific Effects Design

The v1 system gave each theme unique character through specialized CSS effects. V2 preserves and upgrades each:

Forge -- Spark Bursts

V1: ::before pseudo-element with forge-spark-flash keyframes. Multi-layer box-shadows create 4 yellow-orange spark points that appear and spread at two moments during the animation cycle. Only runs on every 3rd particle (--forge-spark-play).

V2: Core embers spawn 2-4 type: 'spark' child particles at ~10% and ~40% lifetime (35% probability). Sparks eject in random directions with high initial velocity (30-80 px/s), heavy drag (4.0), and slight gravity pull. They're colored white-hot to dark-orange via their own sparkColorStops and fade with quadratic falloff (alpha = (1-t)^2). Much more dynamic than v1's fixed box-shadow positions.

Ice -- Diamond Crystals + Twinkles

V1: ::before creates a diamond shape via crossed linear-gradients rotated 45deg, animated with ice-sparkle. ::after creates an expanding circle halo (ice-halo). Base particle has border-radius: 1px (square-ish). Runs on every 4th particle.

V2: Core particles rendered as diamond shapes via _drawDiamond() (4-point polygon). When bright (alpha > 0.3), a cross-hair sparkle is drawn as two thin white rectangles. Separate type: 'twinkle' child particles spawn at ~15% and ~55% lifetime (40% probability) and track their parent crystal's position. Twinkles flash in sharp white pulses using a sawtooth-like pattern and are rendered as 4-point stars via _drawStar4(). Uses IceEmitter subclass for the parent-tracking logic.

Void -- Ghost Split & Orbit

V1: ::before pseudo-element with void-split keyframes. A clone of the particle translates outward to a computed (angle, distance) point, holds, then returns -- simulating a split-and-merge cycle. Uses --void-split-a-x/y CSS variables per particle.

V2: Core particles spawn 1-2 type: 'ghost' child particles at ~18% and ~55% lifetime (50% probability). Ghosts don't use physics forces -- instead they orbit their spawn point on an expanding/contracting sinusoidal path (dist = splitDist * sin(t*PI) -- grows then shrinks). Rendered as a ring (stroked circle) with a bright inner dot, giving an ethereal hollow-orb look distinct from core particles.

Grass -- Spore Puffs

V1: ::before with grass-firefly-glow (pulsing radial glow on every 3rd particle). ::after with grass-dandelion-puff -- 6 box-shadow points arranged hexagonally that expand outward then fade. Uses 6 pairs of --grass-puff-N-x/y CSS variables per particle.

V2: Core particles already blink via oscillating alphaOverLife curves (firefly effect built into the alpha stops). At ~20% and ~55% lifetime, they burst 5-7 type: 'spore' child particles in a radial pattern (30% probability). Spores eject outward evenly-spaced with slight angular jitter, decelerate via drag, and gently lift with negative gravity. Smaller than core particles (0.6x scale). The radial burst replicates the v1 dandelion-puff's hexagonal spread but with actual physics and smooth motion instead of fixed keyframe positions.

Key Design Decisions & Tradeoffs

Why no external libraries?

PixiJS (WebGL) would handle 10x more particles but adds ~400KB, a WebGL context, and a rendering pipeline we'd have to integrate with React's lifecycle. For 200-500 particles, Canvas 2D is more than sufficient and keeps the dependency footprint at zero. If we ever need 5000+ particles, PixiJS is the next step.

Why child particles instead of pseudo-element equivalents?

V1 used ::before/::after pseudo-elements for effects (sparks, twinkles, ghosts, spores). In Canvas 2D there are no pseudo-elements, so we use child particles drawn from the same pool. This is actually more powerful: child particles have real physics, independent lifetimes, and their own color/size interpolation. V1's pseudo-element effects were static relative to their parent; v2's children move independently.

Why linear RGB interpolation instead of CIELAB?

CIELAB produces more perceptually uniform gradients, but requires RGB-to-Lab-to-RGB conversion per particle per frame. For our saturated warm/cool palettes with 5-6 stops, linear RGB looks good and avoids the math overhead. Could revisit if we add themes with problematic hue transitions (e.g., red-to-green).

Why pre-allocated pool instead of array push/filter?

At 60fps with hundreds of particles being born and dying each second, push(new Particle) and filter(p => p.alive) would trigger frequent garbage collection pauses. The pool eliminates all allocation during the animation loop. Measured: pool overhead is ~0.5KB per particle * 600 = ~300KB constant -- negligible.

Tuning philosophy

This demo exposes all the knobs. In production, themes define fixed configs and only the heat multiplier varies at runtime. The sliders here are for finding the right defaults. Once tuned, the slider values become the hardcoded config in particleThemes.js.

Performance: V1 vs V2 -- honest comparison

V1 (CSS/DOM) had one genuine advantage: CSS transform + opacity animations run on the browser's GPU compositor thread. Zero JavaScript per frame for the actual animation. For ~80 particles, that's very efficient.

V2 (Canvas 2D) runs JavaScript every frame: force calculations, noise sampling, color interpolation, and draw calls for every active particle. At 200-400 particles that's fine (~1-2ms/frame), but it's more CPU work than v1 was doing for its 80 particles.

Where V2 wins:

  • Scaling ceiling -- V1 hits a wall around 100 DOM nodes (each with ::before/::after = 240 elements). Adding more causes layout thrashing. Canvas doesn't care -- 200 vs 400 particles is just more arc() calls on the same single element
  • Additive blending -- globalCompositeOperation: 'lighter' is cheap. V1's glow used multi-layer box-shadow, one of the most expensive CSS properties to animate
  • Child particles (sparks, spores, ghosts) -- in V1 these were pseudo-elements with complex box-shadow animations. In V2 they're just more pool entries with simple circle draws

Where V2 is slightly worse:

  • Idle cost -- V1's animation loop was the browser compositor (free). V2 runs a JS callback every 16ms even when nothing visually changes
  • Simple particles -- a basic rising ember with no effects is cheaper as a CSS-animated span than a canvas draw

Net assessment: For the same particle count and complexity, roughly equivalent. The real win is capability (multi-color, physics, child particles, additive blending), not raw throughput at 80 particles.

Production optimization TODO: idle RAF shutdown

The ParticleCanvas component already returns null when heatStage < 3, which prevents canvas creation entirely. But there's an edge case: when heat drops from 3+ to below 3, the RAF loop should be cancelled and the canvas cleared once all remaining particles have finished their lifetimes (not abruptly killed). Implement this as:

// In the RAF loop:
if (heatStage < 3 && emitter.pool.activeCount === 0) {
  cancelAnimationFrame(rafId);
  ctx.clearRect(0, 0, w, h);
  return; // loop stops, zero idle cost
}
// When heatStage drops below 3: stop emitting (rate=0)
// but keep the loop running until activeCount hits 0
// so existing particles fade out gracefully.

This gives the best of both worlds: zero JS cost when particles aren't needed, but graceful fade-out instead of an abrupt cut when heat cools down. The component can also re-enter by starting a new RAF loop when heatStage rises back to 3+.
